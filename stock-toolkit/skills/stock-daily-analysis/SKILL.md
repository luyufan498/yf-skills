# 股票深度分析

执行系统化的股票深度分析流程，通过自适应搜索和多数据源整合，生成专业的股票分析报告。

## 分析模式识别

根据用户请求识别分析模式，选择合适的执行路径：

### 模式A：通用日分析（默认）
- **触发**：用户说"分析XX股票"、"XX股票每日分析"、"帮我分析XX"
- **流程**：执行完整10步流程
- **报告**：完整三层分析框架

### 模式B：问题驱动分析
- **触发**：用户提出具体问题（如"为什么今天大跌"、"为什么这个操作"、"为什么早上买入下午卖出"）
- **流程**：
  1. 步骤1（股价数据）- 必须，重点关注分时数据
  2. 步骤5（模拟盘持仓）- 必须，重点关注 `ptrade operations` 近7日交易记录
  3. 步骤2-4（历史/搜索/广发）- 按需，聚焦问题相关维度
  4. **新增"事件复盘"章节** - 针对用户问题深度分析交易时间线、触发原因、逻辑评估
- **报告**：聚焦问题，可省略无关章节

## 核心流程

分析流程采用并行执行策略，通过多个专用 subagent 同时处理不同维度任务：

- [ ] 获取股价数据 - 实时股价、涨跌幅、K 线趋势 + 标准化股票信息
- [ ] 读取历史分析 - 建立分析连续性和趋势对比
- [ ] 自适应深度搜索 - 多轮迭代收集新闻和热点
- [ ] 获取专业数据 - 龙虎榜、财务基本面、ETF 榜单
- [ ] 获取模拟盘持仓 - 读取当前持仓状态和资金池信息
- [ ] 整合中间数据 - 读取所有 subagent 保存的中间数据，获取交易条件表
- [ ] 条件审查与交易决策 - 依据 trading-discipline.md 审查条件并输出投资建议
- [ ] 执行交易条件更新 - 根据决策更新 conditions 并重新获取最终状态
- [ ] 生成并保存分析报告 - 三层分析框架 + 模拟盘状态 + 结构化存储
- [ ] 输出执行摘要 - 完成状态和关键信息总结

**执行特点**：
- ⚡ **并行执行**：历史、搜索、广发数据三个 subagent 并行工作
- 🔗 **统一管理**：所有 subagent 由主 session 创建，确保完全隔离
- 📊 **多维整合**：历史、新闻、专业数据、股价全面融合


## 执行步骤

### 步骤 1: 获取股价数据

**主 agent 直接执行**：使用 ptrade CLI 工具获取实时股价、K 线数据以及标准化的股票名称和代码。

**执行原则**（主 agent 操作）：
- 主 agent 直接使用 ptrade 命令
- 用户输入的名称可能不准确，进行模糊匹配和验证，确保获取到正确的股票名称和代码，供后续步骤使用
- 获取标准化股票信息后，将其保存为全局变量或传递给后续步骤使用

**执行步骤**（主 agent 直接操作）：
1. 使用 ptrade search 命令搜索股票，获取标准化的股票名称和代码：
   ```bash
   ptrade search "<用户输入>" --limit 5
   ```
2. 根据搜索结果匹配用户意图，确定正确的股票名称和代码（如：sh603019、sz000858）
3. 使用 `ptrade market-summary <股票代码> -f markdown` 获取多周期市场数据（**使用股票代码而非名称，避免中文编码问题**）：
   - **实时股价**：当前价、昨收、涨跌幅
   - **月K数据**：近6个月走势，含每根K线的开/收/高/低/环比涨跌
   - **周K数据**：近8周走势
   - **日K数据**：近5日走势
   - **分时数据**：当日关键时间点（开盘、早盘高点、午盘低点、午后企稳、收盘）及振幅
   - **股票信息**：
     - 完整的股票名称（中文全称）
     - 正确的股票代码（如：600519.SH、000858.SZ）
4. 如数据获取失败，标注"数据暂不可用，建议使用东方财富/同花顺查看"
5. 使用标准化股票名称(简体中文)和代码继续后续步骤

### 步骤 2: 读取历史分析

创建 subagent，要求 subagent 使用 ptrade CLI 工具读取历史分析报告并生成连续性分析。

**执行原则**（主 agent 操作）：
- 创建专门的 subagent
- 在 subagent 的 prompt 中明确要求其智能选择历史分析数据并保存连续性报告

**任务要求**（传递给 subagent 的 prompt 内容）：
1. subagent 读取 `references/analysis_history_guide.md` 了解执行方法
2. 使用 `ptrade analysis <股票名> --action list --limit 30` 列出最近 30 次分析记录
3. **智能选择报告**：根据时间分布选择短期（最近 1-3 天）、中期（一周左右）和长期（近一个月）的代表性报告
4. 读取选定报告并分析预测准确性、观点演进、建议更新
5. **保存连续性分析报告**到 temp-data 目录（使用 `ptrade temp-data --action save --category history-continuity` 命令）

### 步骤 3: 自适应深度搜索

创建 subagent，要求 subagent 执行深度搜索任务收集新闻和热点信息。

**执行原则**（主 agent 操作）：
- 创建专门的搜索 subagent
- 在 subagent 的 prompt 中明确要求其执行深度搜索

**任务要求**（传递给 subagent 的 prompt 内容）：
1. subagent 读取 `references/deep_search_guide.md` 了解详细执行方法
2. 优先使用 searxng-search-skill（本地部署，免费），遇到验证码切换到 brave-search
3. 获取市场新闻：使用 `ptrade fetch-news --source all --limit 30`
4. 按以下维度执行深度搜索：
   - 公司基本面（财务数据、经营指标、业务进展）
   - 行业生态（竞争格局、行业趋势、产业链）
   - 政策环境（行业政策、宏观影响、地区政策）
   - 实时动态（最近 1-2 周的新闻热点）
5. **保存搜索报告**到 temp-data 目录（使用 `ptrade temp-data --action save --category deep-search` 命令）

### 步骤 4: 获取广发证券专业数据

创建 subagent，要求 subagent 调用 gf-finance skill 执行专业数据获取任务。

**执行原则**（主 agent 操作）：
- 创建专门的 subagent
- 在 subagent 的 prompt 中明确要求其调用 gf-finance skill 完成数据获取

**任务要求**（传递给 subagent 的 prompt 内容）：
1. subagent 读取 `references/gf_finance_guide.md` 了解执行指南
2. 调用 gf-finance skill 获取以下三维度数据：
   - **龙虎榜数据**：净买入/卖出、知名席位动向
   - **财务基本面**：PE/PB估值分位、ROE、现金流
   - **ETF 榜单**：相关ETF涨幅、资金流入
3. **保存广发数据汇总**到 temp-data 目录（使用 `ptrade temp-data --action save --category gf-summary` 命令）

### 步骤 5: 获取模拟盘持仓

**主 agent 操作**：通过调用 paper-trading skill 获取当前模拟盘持仓状态和收益情况。

**核心原则**：模拟盘状态表格由 CLI 工具直接生成 Markdown 格式，**agent 只做复制粘贴，不做任何计算**。

**执行步骤**：
1. 如主 agent 已熟悉 ptrade 命令，可直接使用；否则使用 Skill 工具调用 `paper-trading` 了解 ptrade 命令使用方法

2. 尝试调用 `ptrade info "<股票名>" --format markdown` 命令获取 Markdown 格式的模拟盘状态表格：
   - 如果命令成功，CLI 直接返回完整的 Markdown 表格
   - 如果返回"未找到账户"或类似错误，则执行步骤 3

3. 如果账户未初始化，自动初始化资金池（默认 500,000 元）：
   ```bash
   ptrade init "<股票名>" --capital 500000
   ```

4. 重新调用 `ptrade info "<股票名>" --format markdown` 获取表格

5. 调用 `ptrade operations "<股票名>" --days 7` 查看近7日交易记录，核对近期是否已执行过减仓/加仓操作，防止报告给出重复建议。**特别提取以下信息供步骤7防循环检查使用**：
   - 近7日内是否有"卖出"记录的备注中包含"成本保护"/"亏损止损"/"技术破位"/"硬条件触发"等关键词 → 标记为"近期止损事件"
   - 最近一次止损事件的日期、卖出价格、触发原因
   - 止损事件后是否有后续"买入"记录（即"止损后买回"模式）
   - 若存在"止损后买回"，核对买回价格 vs 止损卖出价的偏离（用于规则13成本抬高检查）

6. **零计算规则**：
   - ❌ 禁止 agent 自行计算总盈亏、总收益率、持仓市值等任何数值
   - ✅ 直接将 `ptrade info --format markdown` 的输出粘贴到报告 `## 💰 模拟盘状态` 章节
   - ✅ 如果输出包含"空仓"，按表格实际内容显示即可，无需特殊处理

7. 在表格下方追加近7日操作摘要（来自 `ptrade operations` 的输出）

8. 如果初始化了资金池，在报告中标注"本次分析已创建模拟盘资金池"

**数据整合要点**：
- ✅ 持仓情况将影响投资建议的表述（如空仓可积极建仓，重仓则谨慎）
- ✅ 收益情况用于评估历史操作的有效性
- ✅ 记录当前持仓状态作为报告的"基准状态"
- ✅ 近期操作记录用于校验一致性，避免重复判断加建仓
- ✅ **近期止损事件**是步骤7"防循环硬约束"的关键输入，必须显式提取并传递到决策环节，不得忽略
- ✅ 为后续异步执行交易行为提供基础数据

### 步骤 6: 整合中间数据

**主 agent 操作**：读取所有 subagent 保存的中间数据，并补全写报告和做决策必需的其他前置数据。**步骤6完成后至步骤9开始前，禁止为生成报告而调用数据获取类 CLI 命令（market-summary/fetch-news/info/operations/temp-data read）。步骤7只产生决策，步骤8的条件更新类命令（conditions update/event-set/remove）和更新后的 conditions 查询是允许的。所有数据必须在对应步骤前就绪。**

**数据来源清单**：

| 数据 | 来源 | 获取命令 |
|------|------|---------|
| 历史连续性分析 | Subagent | `ptrade temp-data "<股票名>" --action read --category history-continuity` |
| 深度搜索报告 | Subagent | `ptrade temp-data "<股票名>" --action read --category deep-search` |
| 广发证券数据 | Subagent | `ptrade temp-data "<股票名>" --action read --category gf-summary` |
| 交易条件表 | CLI 直接查询 | `ptrade conditions "<股票名>" --format markdown --template all` |
| 模拟盘状态 | 步骤5已获取 | 复用步骤5的输出 |

**执行步骤**：
1. 读取历史连续性分析
2. 读取深度搜索报告
3. 读取广发证券数据
4. 执行 `ptrade conditions "<股票名>" --format markdown --template all`，获取完整条件输出并保存到变量
5. 确认模拟盘表格数据已就绪
6. 将所有数据整合到变量中，供步骤7审查和步骤9生成报告使用

**数据整合要点**：
- ✅ 确保所有前置数据都已就绪
- ✅ 提取各数据的关键信息
- ✅ 识别数据之间的关联和矛盾
- ✅ 准备好结构化数据供报告生成使用

### 步骤 7: 条件审查与交易决策

基于步骤6整合的数据，先完成交易纪律审查和投资决策，输出明确的投资建议与条件更新清单。**本步骤只产生决策，不修改 conditions。**

**执行前提**：主 agent 已阅读 [trading-discipline.md](references/trading-discipline.md)。

**输出要求**：
1. **条件触发审查结论**
2. **操作建议**（买入/卖出/持有/减仓比例）
3. **本期新条件设定清单**（如需更新 conditions，明确价格、类型、动作）

### 第二步：操作建议

**本期建议**：[一次性建仓（比例） / 买点下沿 X% + 中沿 Y% + 上沿 Z% / 加仓（比例） / 卖出 / 持有 / 减仓（比例）]

> **建仓 vs 加仓语义**：空仓状态下首次建仓使用"买点下沿/中沿/上沿"（区间捕捉）或"一次性建仓"；已有持仓追加使用"加仓"。

**🚫 防循环硬约束（优先级高于"空仓偏离建议"）**：

在给出"买入/加仓/买点建仓"建议前，必须逐条核对以下三条，**任一不满足则禁止建议买入**，改建议"空仓观察"或"持有现有仓位"：

1. **近期止损隔离期**：若近 2 个交易日内发生过止损清仓（成本保护/亏损止损/技术破位触发），本次建议不得为"买入"或"买点建仓"，除非已出现明确的右侧信号（放量突破关键阻力位、MACD金叉确认等）。
2. **成本抬高约束**：若重新建仓成本比上一次止损卖出价抬高 > 1% 且无右侧信号，禁止建议买入或买点建仓（规则13）。
3. **空仓偏离不等于必须买入**：报告建议持有 X 股、当前空仓，**不构成买入理由**。必须先通过上述两条检查，再结合三层分析的核心观点判断是否真的应该建仓。若基本面或短期线索不支持，应明确建议"空仓观察，等待更好的入场点"，并在理由摘要中说明"虽偏离报告建议仓位，但受止损隔离期/成本抬高约束，本期不建仓"。

**🚫 区间捕捉完整性约束**（给出"买点下沿/中沿/上沿"建议时必查）：

1. **多价位覆盖**：若当前空仓且建议区间捕捉建仓，本期必须同时设定 3 个买点价位的 `add_position` 事件（通过 `ptrade conditions --action event-set --event-type add_position`），不得只设 1 个价位。单价位等同于一次性建仓，失去区间捕捉意义。
2. **事件独立性**：多个事件是 OR 关系，任一触发即完成部分建仓。CLI 不维护批次依赖，触发后 agent 需在下一期步骤 8 决定其他未触发事件的处理（移除或保留）。
3. **比例合规**：单事件比例不超过 40%，三事件比例加总等于目标仓位（如 40%+30%+20%=90%，预留 10% 现金缓冲）（规则 3.0.2）。

**理由摘要**（2-3句，结合三层分析和审查结论）：
- 核心观点
- 条件状态匹配度
- 关键催化剂时间窗口

### 步骤 8: 执行交易条件更新

根据步骤7产生的条件更新清单，**执行 CLI 命令修改 conditions**。本步骤完成后，conditions 必须与投资决策一致。

**执行命令**：

```bash
# 示例：更新移动止损
ptrade conditions "<股票名>" --action update --type trailing_stop --price XX.XX --reason "支撑位上调"

# 示例：设定加仓条件（事件条件，已有持仓追加）
ptrade conditions "<股票名>" --action event-set --event-type add_position \
  --price XX.XX --action-str "成本区补仓-加仓30%" --category soft --expiry-days 7

# 示例：设定区间捕捉建仓条件（空仓 → 目标仓位，多个独立事件覆盖买点区间）
# 买点下沿
ptrade conditions "<股票名>" --action event-set --event-type add_position \
  --price 80.00 --action-str "买点下沿-建仓40%" --category soft --expiry-days 7
# 买点中沿
ptrade conditions "<股票名>" --action event-set --event-type add_position \
  --price 81.00 --action-str "买点中沿-建仓30%" --category soft --expiry-days 7
# 买点上沿
ptrade conditions "<股票名>" --action event-set --event-type add_position \
  --price 82.00 --action-str "买点上沿-建仓20%" --category soft --expiry-days 7

# 示例：设定亏损保护梯度（事件条件）
ptrade conditions "<股票名>" --action event-set --event-type loss_protect \
  --price XX.XX --action-str "减仓20%" --category hard

# 示例：移除过期条件
ptrade conditions "<股票名>" --action remove --type take_profit_1

# 示例：移除已触发或需调整的事件条件（如区间捕捉中某价位已触发，移除其他未触发事件）
ptrade conditions "<股票名>" --action event-remove --event-id <event_id>
```

**区间捕捉建仓场景的处理规则**：

1. **事件独立性**：多个 `add_position` 事件是 OR 关系，任一触发即完成部分建仓。CLI 不维护批次依赖，每个事件独立存储和触发。
2. **触发后管理**：若步骤 7 判定"部分买点已触发、已满足目标仓位"，本步骤移除其他未触发的买点事件（`event-remove --event-id <id>` 逐个移除）。若仍想等更低价，可以保留。
3. **调整价位**：若需调整某个买点价位，目前需要 remove + set 两步（先 `event-remove` 移除原事件，再 `event-set` 设定新价位）。
4. **`--action-str` 命名规范**：使用 `<触发位置>-建仓<比例>%` 格式（如"买点下沿-建仓40%"），便于步骤 7 审查时识别。

**执行后检查**：
1. 重新执行 `ptrade conditions "<股票名>" --format markdown --template all`
2. 重新执行 `ptrade conditions "<股票名>" --action event-list`，确认事件条件显示正确（若涉及区间捕捉建仓）
3. 确认所有新增/修改/移除的条件已正确生效
4. 将最终条件输出保存到变量，供步骤9生成报告时使用

**🚫 防循环硬约束（优先级高于"空仓偏离建议"）**：

在给出"买入/加仓/买点建仓"建议前，必须逐条核对以下三条，**任一不满足则禁止建议买入**，改建议"空仓观察"或"持有现有仓位"：

1. **近期止损隔离期**：若近 2 个交易日内发生过止损清仓（成本保护/亏损止损/技术破位触发），本次建议不得为"买入"或"买点建仓"，除非已出现明确的右侧信号（放量突破关键阻力位、MACD金叉确认等）。
2. **成本抬高约束**：若重新建仓成本比上一次止损卖出价抬高 > 1% 且无右侧信号，禁止建议买入或买点建仓（规则13）。
3. **空仓偏离不等于必须买入**：报告建议持有 X 股、当前空仓，**不构成买入理由**。必须先通过上述两条检查，再结合三层分析的核心观点判断是否真的应该建仓。若基本面或短期线索不支持，应明确建议"空仓观察，等待更好的入场点"，并在理由摘要中说明"虽偏离报告建议仓位，但受止损隔离期/成本抬高约束，本期不建仓"。

1. **多价位覆盖**：若当前空仓且建议区间捕捉建仓，本期必须同时设定 3 个买点价位的 `add_position` 事件（通过 `ptrade conditions --action event-set --event-type add_position`），不得只设 1 个价位。单价位等同于一次性建仓，失去区间捕捉意义。

## 注意事项

### 执行流程相关
- **搜索迭代**：根据搜索结果质量调整搜索词，进行深度挖掘
- **数据验证**：如股价数据获取失败，在报告中标注"数据不可用"
- **报告保存**：股票分析完成后立即保存，避免丢失

### 工具使用相关
- **Skill 调用**：股价数据、广发数据、模拟盘交易系统通过创建 subagent 调用对应 SKILL 完成，主 agent 无需感知具体路径
  - paper-trading skill：使用 Skill 工具调用 `paper-trading` 自行读取文档了解 ptrade 命令使用方法
- **数据验证**：如数据获取失败，在报告中标注"数据暂不可用"

### 数据分析相关
- **三层分析框架**：必须包含公司业务、行业趋势、短期线索三个层面
- **明确预测**：涨跌预测不可模棱两可，必须给出明确判断（⬆️/⬇️/➡️）
- **风险提示**：至少列出 3 项主要风险
- **新闻数量**：最新新闻摘要至少 5 条

### Subagent 管理
- **必须等待**：所有 subagent 必须等待任务完成并获取返回结果后再继续
- **中间数据保存**：每个 subagent 完成任务后，必须将结果保存到 ptrade 的 `temp-data/<股票名>/` 目录（通过 `ptrade temp-data` 命令）
- **主 agent 读取**：主 agent 使用 `ptrade temp-data --action read --category <类别>` 命令读取临时数据，替代文件路径读取，保持工具使用统一性
- **主 agent 读取范围**：`references/trading-discipline.md` 是主 agent 生成报告时必须依据的纪律文档，需在执行条件审查前阅读。`references/` 下的其他文档（如 `analysis_history_guide.md`、`deep_search_guide.md`、`gf_finance_guide.md`）是给 subagent 执行时阅读的，主 agent 无需读取，只需在创建 subagent 的 prompt 中指明让 subagent 读取对应文档即可

### 交易纪律约束（防止预测点漂移）

完整规则参见 [trading-discipline.md](references/trading-discipline.md)。本节列出分析报告生成时必须遵守的核心约束。

1. **触发条件必须标注失效日期**：所有软条件必须写清楚失效日期（YYYY-MM-DD），方便后续报告直接判断
2. **过期条件必须列出**：上期已过期但未触发的条件，必须在报告中列出并说明是否进入复审
3. **硬条件只升不降**：止损位一旦上移不得下调；移动止损随利润增加只升不降
4. **除权后条件重置**：如果期间发生除权除息（送转、分红），且条件是在除权前设定的绝对价格条件（止盈、止损、加仓），允许删除旧条件并重新设定。新条件必须基于除权后的股价水平设定，并在报告中声明重置理由（"因送转/分红导致股价基准变化"）。成本保护（auto_link_cost）除外，它由系统自动同步为 成本价×0.985（含1.5%缓冲）。
5. **操作建议非持有化**：条件触发后必须给出对应动作（减仓/加仓/清仓），不得以"基本面未变"为由维持"持有"
6. **利润保护强制**：利润回撤按四档梯度执行，任何一档触发都必须建议对应减仓，不得无视
7. **加仓必须有纪律依据**：任何"买入/加仓"建议必须对应 [trading-discipline.md](references/trading-discipline.md) 中明确的加仓条件类型（成本区/支撑位/急跌/突破），不得仅凭"看好"加仓
8. **原始目标不可篡改**：历史回顾中的目标价位必须是原始设定值，不得事后修改
9. **复审声明强制**：若本期修改了上期条件，必须在报告中声明修改理由和新旧对比
10. **强制决策优先**：生成报告时，先执行「第一步：上期条件状态检查」，根据检查结果决定操作建议，不得跳过条件检查直接给出"持有"
